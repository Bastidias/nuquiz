# NuQuiz Architecture

## Core Principle

Questions are **operations on structured data**, not authored content. Knowledge is modeled as SPO triples. Questions are generated by hiding parts of those triples and sourcing distractors from the same dataset. No LLMs in the generation path — correctness is guaranteed by construction.

See [teacher.md](./teacher.md) for the full question design framework.

---

## Data Model

### Knowledge Layer: SPO Triples

The atomic unit is a **triple**: `(Subject, Predicate, Object)`.

- Each Object is exactly one fact — never compound values
- A Subject-Predicate pair (cell) can have multiple Objects (multiple triples)
- Triples sharing a Predicate form natural comparison sets across Subjects
- Objects within a Predicate are either **shared** (appear for multiple Subjects) or **discriminating** (unique to one Subject)

This atomicity is what makes question generation, distractor sourcing, and per-fact diagnostics possible. See the "Core Principle: One Fact, One Row" section in teacher.md.

### Content Layer: Test Structure

Triples are organized into quizzable content through a hierarchy of **sections** and **topics**. Tags cross-reference triples across the content structure so the same triple can appear in multiple contexts.

The content layer defines how quizzes are assembled — which triples to pull, in what grouping. The knowledge layer stays independent and reusable.

### Storage

- All tables defined via Drizzle ORM — never raw SQL
- Triples stored as normalized rows (subject, predicate, object columns), not as JSON blobs
- Content hierarchy (sections, topics) stored separately and references triples
- Tags stored as a many-to-many relation on triples
- User progress tracked per-triple (see Learning Tracking below)

## Question Generation

Questions are defined by three independent dimensions:

| Dimension | Options |
|-----------|---------|
| **Axis** | Which part of the triple is hidden — Subject, Predicate, or Object |
| **Scope** | How many triples are involved — single, cell, paired, subject profile, cross-subject |
| **Format** | How the student responds — MC, select all, matching, true/false, fill-in-the-blank |

Any axis can combine with any scope and any format. This gives fine-grained difficulty control without authoring separate question banks.

### Difficulty Progression

| Dimension | Easier | Harder |
|-----------|--------|--------|
| **Axis** | Hide Object | Hide Subject | Hide Predicate |
| **Scope** | Single triple | Cell recall | Paired comparison | Subject profile | Cross-subject |
| **Format** | Multiple choice | Select all / Matching | True/False | Fill-in-the-blank |

### Distractors

All distractors are sourced from real data in the dataset:

- **Same Predicate, different Subject** — most plausible, first choice
- **Adjacent Predicates** — less plausible, used when more options are needed
- **Composite distractors** — bundles of mostly-correct triples with one Object swapped

Never invent distractors. If the dataset can't supply enough plausible wrong answers, reduce the number of choices rather than fabricating options.

### Shared vs. Discriminating Objects

The system should track which Objects within a Predicate are shared across Subjects and which are discriminating. This distinction drives specific question types (e.g., "select only the symptoms unique to X") and prevents generating questions where shared Objects are used as distractors for each other.

## Learning Tracking

### Per-Triple Mastery

Track mastery at the individual triple level using SM-2 spaced repetition. Each user-triple pair gets a review card with ease factor, interval, and repetition count.

### Quality Derivation

Derive quality scores from observable signals (correctness + response time) rather than asking users to self-rate. This keeps the quiz flow uninterrupted.

### Hierarchical Rollup

Mastery propagates upward through the content hierarchy. A concept's mastery is the proportion of its triples the user has mastered. This enables statements like "You've mastered 73% of Cell Biology."

### Weak Area Identification

Identify clusters of low-mastery triples to surface targeted review. Look for concepts where mastery is below threshold, and within those, find triples with low ease factors or high failure counts.

### Review Scheduling

Combine two sources into a daily review queue:

1. **Due cards** — triples where the review interval has elapsed, ordered by most overdue
2. **New cards** — unseen triples, introduced at a controlled rate

Interleave review and new cards (e.g., 3 review then 1 new) to balance reinforcement with forward progress.

## Adaptive Progression

The system should follow tutor-like logic when selecting what to test next. Key moves:

| Trigger | Action |
|---------|--------|
| Student misses a specific triple | Drill that triple across varied formats |
| Student masters individual triples for a Predicate | Promote to cell recall, then paired comparison |
| High accuracy in MC format | Re-test same triples as fill-in-the-blank (reduce scaffolding) |
| Student aces discriminating Objects | Test shared Objects — do they know what's common? |
| Student is confident | Present misattributed Objects as True/False |
| Student passes cell recall and comparison | Present composite distractors |

The progression logic should move independently along the axis, scope, and format dimensions based on demonstrated mastery.

## API Design

### Route Groups

| Group | Purpose |
|-------|---------|
| `/auth/*` | OAuth login/logout/session |
| `/subjects/*` | Content hierarchy CRUD (sections, topics, triples) |
| `/study/*` | Quiz sessions and review queue |
| `/progress/*` | Mastery tracking and analytics |
| `/import/*` | Content ingestion |

### Guidelines

- One file per route group in `packages/api/src/routes/`
- All request/response shapes defined as Zod schemas in `packages/shared`, imported by both API and web
- Use Hono RPC to export route types so the web client gets full type inference with zero manual type definitions
- Quiz session responses should never send correct answers to the client before the student answers — validate server-side and return feedback after submission

---

## Content Ingestion

Three paths for getting triple data into the system:

1. **Structured import** — JSON/YAML file matching the import schema. Validate with Zod before persisting.
2. **Builder UI** — Hierarchical editor in the web app for manual authoring. Should enforce the one-fact-one-row rule at the UI level.
3. **AI-assisted** (future) — LLM transforms raw text/notes into structured triples. User always reviews and approves before import. This is the only place an LLM touches the system.

---

## Project Structure

```
packages/
  shared/   — Zod schemas + inferred types (knowledge, study, progress, import)
  api/      — Hono backend (routes, engine, db, middleware)
  web/      — React SPA (Vite, Hono RPC client)
```

The question generation engine lives in `packages/api/src/engine/` and should be organized around the three dimensions (axis, scope, format) rather than around fixed question "types."
