# Kerberos Authentication Flow

**Domain:** 5 — Identity and Access Management &nbsp;|&nbsp; **Pattern:** Ordered &nbsp;|&nbsp; **Tags:** 5.2, 5.6, 4.3
**Status:** draft (SME review pending)

The six-message Kerberos V5 exchange that gets a client from "I have a password" to "I am authenticated to this service, and the service is authenticated to me." Tested extensively on CISSP because almost every cell is a discriminator: which message carries the TGT, which key encrypts which message, which actor (AS vs TGS vs Service) is talking to whom. Cross-tagged with Domain 4 (4.3) because Kerberos is also the canonical single-sign-on network authentication protocol and appears in network-security contexts.

**Layout convention:** rows are messages in sequence. Columns are attributes of each message ordered left → right from least detail (Name) to most detail (Purpose).

| Step | Name | Direction | Content | Encrypted with | Purpose |
|---|---|---|---|---|---|
| 1 | AS_REQ [s1] | Client → AS [s1] | TGT request [s1] | None | Request TGT [s1] |
| 2 | AS_REP [s1] | AS → Client [s1] | TGT<br>TGS session key [s1] | Client long-term key [s1] | Deliver TGT [s1] |
| 3 | TGS_REQ [s1] | Client → TGS [s1] | TGT<br>Authenticator<br>Service principal name [s1] | TGS session key [s1] | Request service ticket [s1] |
| 4 | TGS_REP [s1] | TGS → Client [s1] | Service ticket<br>Service session key [s1] | TGS session key [s1] | Deliver service ticket [s1] |
| 5 | AP_REQ [s1] | Client → Service [s1] | Service ticket<br>Authenticator [s1] | Service session key [s1] | Authenticate to service [s1] |
| 6 | AP_REP [s1] | Service → Client [s1] | Timestamp [s1] | Service session key [s1] | Mutual authentication [s1] |

## Notes

- **Cell convention:** each `<br>`-separated item is one atomic Fact. **No parentheticals in cells.** Message names (`AS_REQ`, `AS_REP`, `TGS_REQ`, `TGS_REP`, `AP_REQ`, `AP_REP`) are the RFC 4120 canonical identifiers and are treated as atomic tokens.
- **Three sub-exchanges, six messages.** Steps 1–2 are the AS exchange (initial authentication, producing a TGT). Steps 3–4 are the TGS exchange (using the TGT to obtain a service-specific ticket). Steps 5–6 are the AP (Application) exchange (using the service ticket to authenticate to the actual service, optionally with mutual authentication back to the client).
- **The Key Distribution Center (KDC).** Both AS and TGS are services run by the KDC. A Kerberos realm has one logical KDC (often replicated for availability), and a single KDC answers both the AS_REQ and TGS_REQ messages. The separation matters because the AS uses the client's long-term key (derived from password) while the TGS uses the TGS session key delivered in Step 2 — minimizing the exposure of the long-term key.
- **Why the long-term key is only used in Step 2.** After Step 2, all encryption uses short-lived session keys (TGS session key for Steps 3–4, service session key for Steps 5–6). This is one of the core security properties of Kerberos: the client's long-term password-derived key never traverses the network and is only used by the client itself to decrypt Step 2's reply.
- **TGT vs service ticket.** The TGT (Ticket-Granting Ticket) is a generic credential issued by the AS that proves the client authenticated once and can be presented to the TGS to obtain service tickets. Service tickets are per-target-service credentials issued by the TGS. TGT lifetime is typically hours to a workday; service tickets are typically shorter.
- **Authenticators vs tickets.** Tickets are reusable within their lifetime — they are cached by the client and re-presented on each service request. Authenticators are single-use timestamped proofs generated by the client at the moment of use to demonstrate *current* possession of the relevant session key. The Authenticator prevents replay of a captured ticket.
- **Step 1 `Encrypted with = None` is a simplification.** RFC 4120 allows AS_REQ to be sent without encryption, but modern Kerberos deployments (and Microsoft Active Directory by default) require pre-authentication, where the AS_REQ includes a timestamp encrypted with the client's long-term key. Without pre-auth, an attacker can probe any username and collect AS_REPs for offline password cracking (the "AS-REP Roasting" attack). The cell reflects the base protocol; the Notes flag pre-auth as the real-world default.
- **Mutual authentication is optional.** Step 6 (AP_REP) is only sent if the client set the `MUTUAL-REQUIRED` flag in AP_REQ. If the client doesn't ask for mutual authentication, the flow ends at Step 5 and the client has no cryptographic evidence that the service is who it claims to be.
- **Clock skew matters.** Authenticators carry timestamps. Kerberos rejects authenticators outside a configured skew window (default five minutes in Windows AD). A large clock drift between client and KDC produces "clock skew too great" errors and is the single most common cause of Kerberos login failures in practice.
- **Out of scope for this Concept:** PKINIT (public-key initial authentication — replaces the long-term key with an X.509 certificate in Step 1–2), constrained delegation and S4U2Self/S4U2Proxy, Kerberoasting and AS-REP Roasting attacks, cross-realm trusts and referrals, Kerberos FAST (Flexible Authentication Secure Tunneling), Kerberos over TCP vs UDP transport behavior. Each deserves its own Concept.
